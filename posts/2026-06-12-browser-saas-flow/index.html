브라우저를 열고 닫을 때까지 — SaaS 완전 실행 지도

2026년 6월 12일
브라우저네트워크렌더링SaaS인증

앱을 열고, 로그인하고, 버튼 몇 번 누르고, 탭을 닫는다. 이 평범한 흐름 안에 DNS 조회, TCP 핸드셰이크, TLS 협상, JIT 컴파일, WebSocket 연결, Garbage Collection이 모두 들어 있다.

이 글은 그 전 과정을 하나의 흐름으로 정리한다.

고정 전제 — 브라우저 멀티프로세스 구조

모든 단계는 아래 구조 위에서 실행된다. 탭이 죽어도 브라우저가 살아있는 이유, JS가 느려도 스크롤이 부드러운 이유가 여기 있다.

프로세스담당
Browser ProcessUI, 탭 관리, 권한, 저장소 접근
Network ProcessDNS, TCP, TLS, HTTP (브라우저와 분리)
Renderer ProcessHTML 파싱, V8(JS), Layout, Paint — 탭당 1개
GPU ProcessComposite, 화면 출력

Renderer Process가 죽으면 해당 탭만 죽는다. Network Process가 분리되어 있어서 JS가 메인 스레드를 블로킹해도 네트워크 요청은 계속 진행된다.

Phase 0 — 브라우저 Cold Start

브라우저 아이콘을 더블클릭한다.

OS → Browser Process 생성
  → 사용자 프로필 로드 (Cookie, localStorage, 확장 프로그램)
  → GPU Process 초기화
  → 이전 세션 복원 여부 확인 (Session Restore)
  → Renderer Process 생성
  → about:blank 렌더링
  → 주소창 포커스

브라우저가 뜨기 전에 이미 여러 프로세스가 생성된다. 확장 프로그램이 많을수록 이 단계가 느리다.

Phase 1 — URL 입력 & 세션 판별

app.example.com을 입력한다.

URL 파싱
  scheme://host:port/path?query#fragment

이전 페이지가 있으면
  → beforeunload 이벤트 발생
  → 통과 시 이전 Renderer에 unload 신호

브라우저 저장소에서 인증 상태 확인
  ├─ 유효한 Session Cookie or Token 있음
  │    → 로그인 페이지 생략, 앱 직행
  └─ 없음 or 만료
       → 로그인 페이지 로드 (Phase 2 진행)

검색어인지 URL인지도 이 시점에 판별한다. 검색어면 기본 검색엔진으로 리다이렉트된다.

Phase 2 — 네트워크 취득

Network Process가 담당한다. URL 하나를 가져오는 데 아래 단계를 전부 통과한다.

브라우저 캐시 계층

Memory Cache     탭 단위, 가장 빠름, 탭 닫으면 소멸
  ↓ Miss
Disk Cache       HDD/SSD, 재방문 시 유효
  ↓ Miss
Service Worker   개발자가 직접 제어하는 캐시
  ↓ Miss
Push Cache       HTTP/2 Server Push, 단명
  ↓ Miss → 네트워크로

만료된 캐시가 있으면 서버에 조건부 요청을 보낸다.

  • ETag → If-None-Match
  • Last-Modified → If-Modified-Since
  • 304 응답 → 캐시 재사용 / 200 응답 → 새 데이터로 교체

DNS 조회

IP 주소 없이 TCP를 열 수 없다.

브라우저 DNS 캐시 (TTL ~60초)
  ↓ Miss
OS DNS 캐시 (/etc/hosts 포함)
  ↓ Miss
Recursive Resolver (ISP or 8.8.8.8)
  ↓ Miss
Root NS → TLD NS (.com) → Authoritative NS
  → IP 반환 + TTL 저장

CNAME 체인이 있으면 조회 왕복 수가 늘어난다. DNS 조회 자체에서 수십 ms가 소모될 수 있다.

HSTS

HSTS Preload 목록에 등록된 도메인은 HTTP로 요청이 들어오면 브라우저가 네트워크 요청 없이 내부에서 307로 HTTPS로 전환한다. 네트워크 탭에 표시되지 않는 이유가 이것이다.

TCP 3-Way Handshake

SYN  →
     ← SYN-ACK
ACK  →

1 RTT 고정 비용. Connection Pool에 동일 host:port 연결이 있으면 생략된다. HTTP/3(QUIC)는 UDP 기반이라 이 단계가 없다.

TLS 1.3 협상

ClientHello (지원 암호화 목록) →
← Certificate + ServerHello
대칭키 생성 →

TLS 1.3은 1 RTT. 재연결 시 0-RTT Session Resumption으로 핸드셰이크를 생략할 수 있다. TLS 1.2는 2 RTT. OCSP Stapling이 없으면 CA 서버에 인증서 유효성 확인 요청이 별도로 발생한다.

HTTP 버전별 동작 차이

버전연결특징병목
HTTP/1.1도메인당 최대 6개 병렬요청 직렬 처리Head-of-line Blocking
HTTP/21개 연결 + 스트림 멀티플렉싱HPACK 헤더 압축TCP 레벨 HoL Blocking
HTTP/3UDP(QUIC) 기반TCP 핸드셰이크 없음없음

CORS Preflight

cross-origin API 요청이면 실제 요청 전에 OPTIONS 요청이 먼저 나간다.

OPTIONS /api/data →
← Access-Control-Allow-Origin
← Access-Control-Allow-Methods

본 요청 →

Simple Request(GET, POST + 단순 헤더)는 Preflight 없이 바로 전송된다. Preflight 실패 시 브라우저가 요청을 차단한다. 서버는 이미 OPTIONS를 수신했다는 점에 주의.

패킷 전송

HTTP 요청이 MTU(1500 bytes) 단위로 TCP 세그먼트로 분할된다. ISP 백본에서 BGP 라우팅으로 최적 경로를 탐색한다. 패킷 손실이 발생하면 TCP 재전송이 일어나고 추가 RTT가 소모된다.

Phase 3 — 서버 처리

CDN & 보안

Cloudflare / CloudFront 엣지 도달
  WAF: SQL Injection, XSS, OWASP Top 10 필터링
  DDoS: SYN Flood, UDP Flood 차단
  ├─ 정적 캐시 HIT → 즉시 응답 (Phase 4로)
  └─ Miss → 오리진 서버로

Load Balancer & Reverse Proxy

L4 LB (TCP/IP 레벨) → L7 LB (HTTP 레벨)
  분배 알고리즘: Round Robin / Least Connection / IP Hash

Reverse Proxy (NGINX)
  TLS Termination → 이후 평문 HTTP로 내부 통신
  Rate Limiting
  Connection Buffer (WAS 과부하 방지)

TLS Termination이 여기서 일어나기 때문에 WAS는 암호화를 신경 쓰지 않아도 된다.

API Gateway (SaaS 특화)

JWT 서명 검증
Rate Limiting (사용자/플랜별)
마이크로서비스 라우팅

WAS & DB

DispatcherServlet → Filter/Interceptor → HandlerMapping
→ Controller → Service (트랜잭션 시작) → Repository

Cache (Redis)
  HIT → 즉시 반환
  MISS → Query Optimizer → Index Scan or Full Scan + Disk I/O

응답 생성
  JSON 직렬화 → gzip/brotli 압축
  Cache-Control / ETag 헤더 설정

N+1 쿼리는 Repository에서 발생한다. 연관 엔티티를 루프마다 별도 쿼리로 조회하는 패턴이다. Fetch Join으로 해결한다.

Phase 4 — 응답 수신 & 렌더링

Network Process가 Renderer Process에 데이터를 전달한다.

브라우저 캐시 저장

응답 헤더의 Cache-Control 정책에 따라 Memory 또는 Disk Cache에 저장된다.

HTML 파싱

Main Thread: HTML → DOM 순차 생성
Preload Scanner: 별도 스레드로 병렬 실행
  <script>, <link>, <img> 미리 발견
  → 우선순위 큐 등록 → 병렬 다운로드 시작

<script> 태그를 만나면 파싱이 중단된다.

속성동작
없음파싱 중단 (블로킹)
deferDOM 완성 후 실행, 순서 보장
async다운로드 완료 즉시 실행, 순서 미보장

CSSOM & Render Tree

CSS 다운로드 → CSSOM 생성. CSSOM이 완성되기 전까지 Render Tree를 만들 수 없다. JS도 CSSOM을 기다린다. CSS가 JS를 간접 차단하는 이유다.

DOM + CSSOM → Render Tree (display:none 노드 제외)

Layout → Paint → Composite

Layout (Reflow)      크기/위치 계산 — Main Thread
Paint (Repaint)      픽셀 채우기, 레이어별 분리 — Main Thread
Composite            레이어 합성 → GPU Process 전달
GPU 출력             화면 표시

will-change, transform, video, canvas는 별도 Compositing Layer로 승격된다. FCP(First Contentful Paint)는 여기서 측정된다.

Phase 5 — JS 엔진 & Event Loop

V8 실행 파이프라인

소스코드
→ Parser → AST 생성
→ Ignition (인터프리터) → Bytecode 실행
→ Hot Path 감지 (자주 실행되는 코드)
→ TurboFan (JIT 컴파일러) → 기계어 최적화
→ 타입 가정이 깨지면 Deoptimization → Bytecode 복귀

JIT 최적화가 깨지는 대표적인 케이스는 같은 함수에 다른 타입을 넘기는 것이다. add(1, 2) → add('a', 'b')처럼.

Garbage Collection

Minor GC (Scavenger)    Young Generation, 자주/빠르게 실행
Major GC (Mark-Sweep)   Old Generation, 느리고 Heavy

GC가 실행되는 동안 Main Thread가 일시 정지된다. 화면이 뚝뚝 끊기는 Jank의 원인이다.

Event Loop

┌──────────────────────────────┐
│  Call Stack 비어있는지 확인   │◀──────────┐
└─────────────┬────────────────┘           │
              ▼                            │
  Microtask Queue 전부 소진              루프
  (Promise, MutationObserver)              │
              ▼                            │
  rAF Queue (requestAnimationFrame)        │
              ▼                            │
  Task Queue 1개 꺼내서 실행 ─────────────┘
  (setTimeout, 이벤트 핸들러, I/O)

Microtask는 Task 사이마다 전부 소진된다. Promise 체인이 길면 Task가 뒤로 밀린다.

Phase 6 — 인증 처리

이메일/비밀번호

POST /auth/login
→ 서버: 자격증명 검증
→ 응답: Set-Cookie: session=xxx; HttpOnly; Secure; SameSite=Lax
→ 브라우저: Cookie 저장

HttpOnly가 붙으면 JS에서 document.cookie로 접근할 수 없다. XSS 공격으로 토큰을 탈취하지 못하게 하기 위해서다.

OAuth 2.0 / OIDC (SSO)

클릭
→ SaaS 서버 → 302 Redirect to accounts.google.com
→ Google 로그인 (별도 도메인 → Phase 2 DNS/TCP/TLS 다시 수행)
→ 인증 완료 → /callback?code=xxx 로 리다이렉트
→ SaaS 서버: code → AccessToken 교환 (서버 to 서버)
→ Set-Cookie 세션 발급

code는 일회용이고 수명이 짧다. 브라우저에 AccessToken이 직접 노출되지 않는다.

JWT

응답 body: { accessToken, refreshToken }

accessToken  → memory or localStorage
               (localStorage: XSS 취약 / memory: 탭 닫으면 소멸)
refreshToken → HttpOnly Cookie

이후 모든 API 요청: Authorization: Bearer <accessToken>

CSRF 방어는 SameSite=Lax Cookie 또는 X-CSRF-Token 헤더로 처리한다.

Phase 7 — 앱 초기화

Code Splitting

초기 번들: 라우터 + 인증 모듈만 (최소화)
현재 라우트 코드: Dynamic Import로 로드
나머지: 사용자가 이동할 때 Lazy Load

번들 전체를 한 번에 다운로드하면 TTI(Time to Interactive)가 늦어진다. 필요한 코드만 먼저 보낸다.

앱 초기화 순서

저장소에서 토큰 읽기
→ /api/me 호출 (Phase 2~3 전체 통과)
→ 전역 상태 초기화 (Redux / Zustand)
→ 클라이언트 라우터 마운트
→ Skeleton UI 표시
→ 데이터 수신 → 실제 컴포넌트로 교체
→ LCP 측정 시점

Phase 8 — WebSocket 연결

Slack, Figma, Notion처럼 실시간 협업이 있는 SaaS는 HTTP 요청-응답 외에 WebSocket 연결을 상시 유지한다.

연결 수립

GET /ws HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: xxx

← 101 Switching Protocols

기존 TCP 연결 위에서 HTTP 프로토콜을 WebSocket으로 업그레이드한다. 이후 HTTP 헤더 없이 프레임 단위로 통신한다.

연결 유지 & 재연결

Heartbeat: ping/pong 30초 간격
서버 Push → Event Loop Task Queue → 핸들러 실행

연결 끊김 시 Exponential Backoff 재연결
  1s → 2s → 4s → 8s → 최대 30s

Phase 9 — 클릭 & 인터랙션

이벤트 전달 경로

GPU Process: hit-test로 클릭 좌표 판별
→ Compositor Thread: 스크롤 등 처리
→ Renderer Main Thread: 이벤트 전달

이벤트 캡처링: document → target
이벤트 버블링: target → document
→ 핸들러 → Task Queue 등록 → Event Loop → Call Stack

SPA 페이지 이동

History API pushState → URL만 변경 (서버 요청 없음)
→ 클라이언트 라우터: 이전 컴포넌트 unmount
→ 새 컴포넌트 mount → Skeleton UI
→ 데이터만 fetch (Phase 2~3 반복)
→ Virtual DOM diff → 변경 노드만 DOM 패치
→ 부분 Reflow/Repaint → Composite → GPU

전체 페이지를 다시 받는 MPA와 달리 변경된 부분만 갱신한다.

API 요청 & 응답 처리

Optimistic Update
  서버 응답 전에 UI 먼저 반영
  실패 시 롤백

Token 만료 확인
  → Silent Refresh
    POST /auth/refresh (refreshToken Cookie 자동 전송)
    → 새 accessToken 발급 → 원래 요청 재시도

API 요청 전송 (Phase 2~3 전체 통과)
응답 코드처리
200상태 업데이트 → DOM 패치
304캐시 재사용
401Silent Refresh → 재시도 → 실패 시 로그인 페이지
429Retry-After 헤더 읽기 → 대기 후 재시도
5xxExponential Backoff 3회 → 에러 UI
Network ErrorService Worker 캐시 응답 → 오프라인 큐 적재

Reflow 루프

DOM 변경 후 렌더링이 다시 시작된다.

JS → DOM 변경 → Reflow(Layout 재계산) → Repaint → Composite

getBoundingClientRect() 직후 DOM을 변경하면 브라우저가 강제로 Layout을 동기 실행한다. 이 패턴이 반복되면 Reflow가 수십 회 발생한다(Layout Thrashing).

transform/opacity만 변경하면 Layout과 Paint를 건너뛰고 Composite만 실행된다. GPU가 처리하므로 Main Thread를 건드리지 않는다.

  ws.close(1000, 'user closed')

  Memory Cache 소멸
  JS Heap 해제
