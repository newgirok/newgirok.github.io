RSC를 실제로 써보니 — 기대와 현실

2026-06-01
reactrscnext.js

Next.js 13이 App Router를 들고 나왔을 때, React Server Components 얘기를 처음 제대로 접했다.

"컴포넌트가 서버에서 렌더링된다. 데이터 fetching을 컴포넌트 안에서 바로 할 수 있다. 번들 사이즈가 줄어든다."

읽으면서 솔직히 흥분했다. 클라이언트에서 useEffect + useState로 데이터 가져오던 그 패턴이 얼마나 지저분했는지. 로딩 상태 관리, 에러 처리, 경쟁 조건... 매번 같은 코드를 반복했으니까.

근데 막상 프로젝트에 들이붓고 보니.

가장 먼저 막힌 것 — use client 경계

이론상으로는 명확하다. 서버 컴포넌트는 서버에서, 클라이언트 컴포넌트는 브라우저에서. use client를 파일 상단에 적으면 그 컴포넌트부터 아래는 다 클라이언트.

근데 실제로 작업하다 보면 이 경계가 계속 애매해진다.

레이아웃을 서버 컴포넌트로 짜고, 그 안에 상태가 필요한 드롭다운 하나를 넣으려고 했다. 드롭다운 컴포넌트에 use client 달았다. 여기까진 괜찮다.

문제는 그 드롭다운에 서버에서 가져온 데이터를 props로 내려줬을 때다. 서버 컴포넌트 → 클라이언트 컴포넌트로 props를 넘기는 건 되는데, 넘어가는 값이 직렬화 가능해야 한다. 클래스 인스턴스나 함수는 안 된다.

"그래서 뭘 내려보내면 안 된다는 거야?"를 파악하는 데 생각보다 시간이 걸렸다.

Context Provider가 서버 컴포넌트가 될 수 없다

이게 제일 당황스러웠다.

전역 상태를 Context로 관리하는 패턴을 쭉 써왔는데, Provider는 무조건 use client여야 한다. useState나 useContext를 쓰기 때문에. 그러면 <ThemeProvider>나 <AuthProvider>를 루트 레이아웃에 감싸면 그 아래 모든 자식도 클라이언트 컴포넌트가 되는 건가?

아니다. 서버 컴포넌트를 클라이언트 컴포넌트의 children으로 넘기면 여전히 서버에서 렌더링된다. 이 부분을 이해하고 나서야 구조가 잡혔다.

// Provider는 클라이언트 컴포넌트
'use client'
export function Providers({ children }: { children: React.ReactNode }) {
  return <ThemeProvider>{children}</ThemeProvider>
}

// layout.tsx는 서버 컴포넌트
export default function RootLayout({ children }) {
  return (
    <Providers>
      {children} {/* 여기 들어오는 page.tsx는 여전히 서버 컴포넌트 */}
    </Providers>
  )
}

children으로 넘기는 패턴이 핵심이었다.

서드파티 라이브러리 호환성

이건 진짜 현실적인 문제다.

차트 라이브러리, 에디터, 일부 UI 컴포넌트들이 내부적으로 window나 document에 접근한다. 서버에서 실행되면 바로 터진다.

use client 붙이면 해결되긴 하는데, 그러면 그 컴포넌트를 서버에서 데이터와 같이 렌더링하려던 목적이 흐릿해진다.

아직까지 생태계가 완전히 따라잡지 못한 느낌이다. 라이브러리 README에 "RSC compatible"이라고 명시된 게 아니면 일단 의심하고 들어가야 한다.

그래서 RSC가 진짜 빛나는 순간은

불평만 했는데, 실제로 좋았던 케이스를 얘기하자면.

데이터가 많고 인터랙션이 거의 없는 페이지. 블로그 포스트 목록, 대시보드 통계 카드, 문서 페이지 같은 것들.

// 이게 그냥 된다. useEffect 없음. loading 상태 없음.
export default async function PostList() {
  const posts = await db.post.findMany({ orderBy: { date: 'desc' } })
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>)}</ul>
}

컴포넌트 자체가 async 함수다. await 그냥 쓰면 된다. 이게 처음엔 어색했는데 익숙해지니까 편하다.

waterfall 문제도 줄었다. 클라이언트에서 데이터 여러 개 병렬로 가져오려면 Promise.all이니 뭐니 신경 써야 했는데, 서버에서 다 해결하고 HTML로 내려보내니까.

RSC를 도입하면서 제일 크게 바뀐 건 "이 컴포넌트가 서버에서 실행되어야 하나, 브라우저에서 실행되어야 하나"를 먼저 생각하게 됐다는 거다.

예전엔 그냥 다 클라이언트였으니까 그런 생각 자체를 안 했다.

불편함도 있고 삽질도 있었는데, 그 고민 자체가 아키텍처를 더 명확하게 만든 것 같기도 하다.
