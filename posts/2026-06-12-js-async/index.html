자바스크립트 비동기, 한 번에 정리

2026-06-12T20:00:00
JavaScript비동기Promiseasync-await

자바스크립트는 싱글 스레드입니다. 한 번에 하나의 일만 합니다. 그런데 어떻게 서버 요청을 보내면서 동시에 화면도 그릴 수 있을까요.

동기 vs 비동기

동기 (Synchronous)비동기 (Asynchronous)
실행 방식작업이 끝나야 다음으로 넘어감작업을 맡기고 바로 다음으로 넘어감
장점순서가 명확하고 예측 쉬움대기 없이 여러 작업 처리 가능
단점오래 걸리는 작업이 전체를 막음실행 순서 파악이 어려울 수 있음
예계산, 조건문, 반복문서버 요청, 타이머, 파일 읽기

오래 걸리는 작업(네트워크 요청, 타이머)을 동기로 처리하면 그 시간 동안 화면이 완전히 멈춥니다. 비동기는 이 문제를 해결하기 위해 존재합니다.

이벤트 루프 — 비동기의 실체

자바스크립트 엔진은 싱글 스레드지만, 브라우저가 Web APIs를 통해 오래 걸리는 작업을 대신 처리합니다.

  ┌─────────────────┐   async request   ┌─────────────────┐
  │   Call Stack    │ ────────────────▶ │    Web APIs     │
  │  (single thread)│                   │ (multi thread)  │
  └────────┬────────┘                   │ setTimeout      │
           │                            │ fetch           │
           │                            │ DOM Events      │
           │                            └────────┬────────┘
           │                                     │ callback
           │                                     ▼
           │                            ┌─────────────────┐
           │         Event Loop         │ Microtask Queue │  Promise
           │    (Call Stack 비면        ├─────────────────┤
           └◀───꺼내서 실행)─────────── │ Macrotask Queue │  setTimeout
                                        └─────────────────┘
  • Call Stack: JS 코드가 실행되는 곳. 싱글 스레드로 한 번에 하나만 처리
  • Web APIs: 브라우저가 멀티 스레드로 처리하는 영역. 완료되면 콜백을 Queue로 전달
  • Microtask Queue: Promise .then / .catch 콜백이 쌓이는 곳. 우선순위 높음
  • Macrotask Queue: setTimeout, setInterval 콜백이 쌓이는 곳
  • Event Loop: Call Stack이 비면 Microtask를 먼저 모두 소진한 뒤 Macrotask에서 하나씩 꺼냄

실행 순서 3단계

  1. Call Stack에 있는 코드를 모두 실행한다
  2. Microtask Queue(Promise)를 모두 비운다
  3. Macrotask Queue(setTimeout)에서 하나 꺼내 실행한다 — 1번으로 돌아간다

이 규칙 때문에 setTimeout(fn, 0)이어도 Promise보다 늦게 실행됩니다.

setTimeout(() => console.log('A'), 0)   // Macrotask
Promise.resolve().then(() => console.log('B'))  // Microtask
console.log('C')  // 동기

// 출력 순서: C → B → A

비동기 처리의 3단 진화

1단계 — 콜백 (Callback)

비동기 작업이 완료됐을 때 실행할 함수를 직접 넘깁니다.

function getUser(id, callback) {
  setTimeout(() => callback({ id, name: '김철수' }), 1000)
}

// 작업이 중첩될수록 오른쪽으로 밀린다
getUser(1, (user) => {
  getPosts(user.id, (posts) => {
    getComments(posts[0].id, (comments) => {
      console.log(comments)
      // 더 중첩되면 감당이 안 된다
    })
  })
})
장점단점
구조가 단순하다중첩이 깊어질수록 가독성이 극도로 나빠진다
에러 처리를 각 단계마다 직접 해야 한다
실행 순서 파악이 어렵다

2단계 — Promise

비동기 작업의 결과를 나타내는 객체입니다. 작업이 성공했는지 실패했는지를 상태로 관리합니다.

        ┌── fulfilled (성공) ──▶ .then(결과 처리)
Pending ─┤
        └── rejected  (실패) ──▶ .catch(에러 처리)
                                      ↓
                               .finally() — 항상 실행

function getUser(id) {
  return new Promise((resolve, reject) => {
    setTimeout(() => resolve({ id, name: '김철수' }), 1000)
  })
}

getUser(1)
  .then(user => getPosts(user.id))   // Promise를 반환하면 체이닝 가능
  .then(posts => getComments(posts[0].id))
  .then(comments => console.log(comments))
  .catch(error => console.error(error))  // 어디서든 에러를 여기서 잡음
  .finally(() => console.log('완료'))

중첩 대신 체이닝으로 평탄해지고, 에러 처리도 .catch() 하나로 통일됩니다.

여러 요청을 동시에 보낼 때

// 순차 실행 — 1초 + 1초 = 2초 소요
const user = await getUser(1)
const config = await getConfig()

// 병렬 실행 — 가장 느린 것 기준 (둘 다 1초면 1초)
const [user, config] = await Promise.all([getUser(1), getConfig()])
메서드동작실패 처리
Promise.all전부 성공해야 결과 반환하나라도 실패하면 전체 실패
Promise.allSettled성공/실패 관계없이 모두 완료각각 결과를 배열로 반환
Promise.race가장 먼저 완료된 것만 반환가장 먼저 실패한 것이 기준

3단계 — async / await

Promise를 동기 코드처럼 읽히게 써주는 문법입니다. 실제 동작은 Promise와 동일합니다.

// Promise 방식
getUser(1)
  .then(user => getPosts(user.id))
  .then(posts => getComments(posts[0].id))
  .then(comments => console.log(comments))
  .catch(console.error)

// async/await 방식 — 같은 동작, 더 읽기 쉬움
async function loadData() {
  try {
    const user = await getUser(1)
    const posts = await getPosts(user.id)
    const comments = await getComments(posts[0].id)
    console.log(comments)
  } catch (error) {
    console.error(error)
  }
}

await가 블로킹이 아닌 이유

await를 만나면 해당 async 함수가 일시 중단되고, Call Stack에서 제거됩니다. 메인 스레드는 다음 코드를 계속 실행합니다.

async function loadData() {
  console.log('1: 함수 시작')
  const user = await getUser(1)   // 여기서 함수가 일시 중단
  console.log('3: user 받음')     // Promise 완료 후 Microtask로 재개
}

loadData()
console.log('2: 이건 loadData가 멈춘 사이에 실행됨')

// 출력: 1 → 2 → 3

await getUser(1)을 만나는 순간 일어나는 일을 단계별로 보면:

단계무슨 일이 일어나나
1getUser(1) 호출 → Promise 반환
2async 함수 일시 중단, await 이후 코드를 Promise의 .then() 콜백으로 등록 (Microtask)
3loadData 함수 프레임이 Call Stack에서 제거 → 메인 스레드 해방
4console.log('2') 실행
5Web API에서 getUser() 완료 → Microtask Queue에 콜백 등록
6Call Stack이 비면 Event Loop가 Microtask를 꺼내 실행 → console.log('3') 재개

await는 함수를 중단하는 것이지 스레드를 중단하는 것이 아닙니다. Call Stack이 즉시 비워지기 때문에 다른 코드가 그 사이에 실행될 수 있습니다.

실전 — Fetch API

// GET 요청
async function getPost(id) {
  const response = await fetch(`/api/posts/${id}`)
  if (!response.ok) throw new Error(`HTTP ${response.status}`)  // 4xx/5xx 직접 체크
  return response.json()
}

// POST 요청
async function createPost(data) {
  const response = await fetch('/api/posts', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(data),
  })
  return response.json()
}

fetch()는 네트워크 자체가 끊기지 않으면 reject하지 않습니다. 404나 500이 와도 .catch()가 실행되지 않습니다. response.ok로 직접 확인해야 합니다.

세 가지 방식 비교

콜백Promiseasync/await
가독성낮음 (중첩)중간 (체이닝)높음 (동기식 흐름)
에러 처리각 단계마다.catch() 하나try/catch
병렬 처리직접 조율 필요Promise.allawait Promise.all
현재 표준레거시보조 사용권장

두 가지만 기억하면 비동기 코드의 실행 순서가 보입니다.

첫째, await는 함수를 중단하는 것이지 스레드를 중단하는 것이 아닙니다. Call Stack에서 함수가 빠지기 때문에 다른 코드가 그 사이에 실행됩니다. 이벤트 루프의 Microtask Queue가 함수의 재개 시점을 관리합니다.

둘째, Microtask(Promise)는 Macrotask(setTimeout)보다 항상 먼저 실행됩니다. Event Loop는 Call Stack이 비면 Microtask를 전부 소진한 뒤에야 Macrotask로 넘어갑니다.
