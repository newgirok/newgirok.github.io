로그인 버튼을 눌렀다. 화면에 아무것도 변하지 않는 0.3초가 있다. 그 사이에 무슨 일이 벌어지고 있을까.

## 요청의 여정

버튼 클릭 하나가 실제로 거치는 경로입니다.

```
[사용자]
   │ 버튼 클릭
   ▼
[브라우저 — JS]
   │ fetch('https://api.example.com/login', { id, password })
   ▼
[네트워크]
   │ HTTP 요청 패킷 전송
   ▼
[서버]
   │ 요청 수신 → 비밀번호 검증
   ▼
[데이터베이스]
   │ 유저 정보 조회
   ▼
[서버]
   │ 응답 생성 → { token: '...' }
   ▼
[네트워크]
   │ HTTP 응답 패킷 전송
   ▼
[브라우저 — JS]
   │ 응답 수신 → 화면 업데이트
   ▼
[사용자]
   로그인 완료
```

이 전체 과정이 보통 100ms~1초 사이에 일어납니다. 사용자 입장에서는 찰나지만, 컴퓨터 입장에서는 굉장히 긴 시간입니다.

## 만약 이걸 순서대로 기다린다면

먼저 용어부터 짚고 넘어갑니다.

**스레드(Thread)** 는 작업을 처리하는 일꾼 하나라고 생각하면 됩니다. 자바스크립트는 이 일꾼이 한 명입니다. 즉, 한 번에 하나의 일만 할 수 있습니다.

**동기(Synchronous)** 는 "동시에 같이"가 아니라 "순서를 맞춰서"라는 뜻입니다. 요청을 보내면 응답이 올 때까지 다음 작업을 시작하지 않고 기다립니다.

**비동기(Asynchronous)** 는 반대입니다. 요청을 보내놓고 기다리지 않습니다. 응답이 오면 그때 처리합니다. 그 사이에 다른 일을 계속합니다.

자바스크립트 일꾼(스레드)이 한 명인데 동기 방식으로 서버 응답을 기다린다면 어떻게 될까요.

```
버튼 클릭
   → 서버에 요청 전송
   → ........일꾼이 여기 멈춰서 기다림........ (500ms)
   → 응답 수신
   → 화면 업데이트
```

이 500ms 동안 브라우저는 아무것도 하지 못합니다. 일꾼이 응답만 기다리고 있기 때문입니다. 스크롤도, 다른 버튼 클릭도, 애니메이션도 전부 멈춥니다. 화면이 얼어붙습니다.

## 비동기가 해결하는 것

비동기는 "기다리는 동안 다른 일을 한다"는 방식입니다.

```
버튼 클릭
   → 서버에 요청 전송
   → (요청은 브라우저가 백그라운드에서 처리)
   → JS는 다른 작업 계속 진행  ← 화면 멈추지 않음
   → 응답이 오면 그때 처리
```

네트워크 요청이나 타이머처럼 "기다려야 하는 작업"은 브라우저의 Web APIs가 대신 맡습니다. JS는 그 작업을 위임하고 다음 코드로 바로 넘어갑니다. 응답이 도착하면 JS가 다시 불려와 처리합니다.

이것이 **비동기(Asynchronous)** 처리입니다.

## 브라우저 안에서 실제로 어떻게 동작하나

JS 엔진 혼자서는 비동기를 할 수 없습니다. 브라우저 전체가 협력해서 가능합니다.

```
  ┌─────────────────┐   async request   ┌─────────────────┐
  │   Call Stack    │ ────────────────▶ │    Web APIs     │
  │  (single thread)│                   │ (multi thread)  │
  └────────┬────────┘                   │ fetch, timer,   │
           │                            │ DOM Events      │
           │                            └────────┬────────┘
           │                                     │ done → callback
           │                                     ▼
           │                            ┌─────────────────┐
           │         Event Loop         │ Microtask Queue │  Promise
           │    (Call Stack 비면        ├─────────────────┤
           └◀───꺼내서 실행)─────────── │ Macrotask Queue │  setTimeout
                                        └─────────────────┘
```

- **Call Stack**: JS 코드가 실행되는 곳. 싱글 스레드
- **Web APIs**: 브라우저가 별도 스레드로 처리하는 영역. fetch, setTimeout, 이벤트 리스너 등
- **Queue**: 작업이 완료되면 콜백 함수가 여기 줄을 섭니다
- **Event Loop**: Call Stack이 비면 Queue에서 콜백을 꺼내 실행합니다

로그인 버튼 클릭의 흐름을 여기에 대입하면:

| 단계 | 어디서 | 무슨 일 |
|---|---|---|
| 1 | Call Stack | `fetch('/login')` 호출 |
| 2 | Web APIs | 네트워크 요청을 브라우저가 대신 처리 |
| 3 | Call Stack | JS는 다음 코드 실행 (화면 유지) |
| 4 | Web APIs → Queue | 서버 응답 도착 → 콜백이 Queue로 이동 |
| 5 | Event Loop | Call Stack 비었으면 콜백을 꺼내 실행 |
| 6 | Call Stack | 응답 처리 → 화면 업데이트 |

## 개발자는 이걸 코드로 어떻게 다루나

비동기 처리 방식은 세 단계로 발전해 왔습니다.

**콜백**: "완료되면 이 함수를 실행해줘"

```javascript
fetch('/login', options, function(response) {
  // 응답 왔을 때 실행
  updateUI(response)
})
```

작업이 중첩되면 코드가 오른쪽으로 계속 밀립니다. 읽기 어려워집니다.

**Promise**: "작업의 결과를 나중에 줄게. 성공하면 .then, 실패하면 .catch"

```javascript
fetch('/login', options)
  .then(response => response.json())
  .then(data => updateUI(data))
  .catch(error => showError(error))
```

콜백 중첩 대신 체이닝으로 평탄해집니다.

**async/await**: "기다리는 것처럼 쓰되, 실제론 비동기로 동작"

```javascript
async function login() {
  try {
    const response = await fetch('/login', options)
    const data = await response.json()
    updateUI(data)
  } catch (error) {
    showError(error)
  }
}
```

`await`를 만나면 함수가 일시 중단되고 Call Stack에서 빠집니다. 브라우저는 그 사이 다른 일을 합니다. 응답이 오면 중단된 지점부터 다시 시작합니다. 코드는 동기처럼 읽히지만 화면은 멈추지 않습니다.

---

버튼 하나를 눌렀을 때 일어나는 일의 핵심은 "기다림을 어떻게 처리하느냐"입니다. 동기로 기다리면 화면이 멈추고, 비동기로 맡기면 브라우저가 계속 살아있습니다. 자바스크립트가 싱글 스레드임에도 웹이 매끄럽게 동작하는 이유가 여기에 있습니다.
