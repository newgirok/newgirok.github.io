자막이 입술보다 0.5초 늦게 나오는 영상을 보면서 "이 정도면 그냥 쓸 수 있지 않나?" 하고 넘어가려던 적이 한두 번이 아니다. 결국 못 넘어갔다.

## 문제의 시작: edge-tts VTT는 단어 단위다

edge-tts가 뱉는 VTT 파일은 단어 하나하나에 타임스탬프가 붙는다. 문장 경계가 없다. 그러니 "안녕하세요 저는" 같은 텍스트가 단어 3개짜리 엔트리 3개로 쪼개져 나온다.

이걸 그냥 SRT로 변환하면? 자막이 단어마다 깜빡거린다. 쓸 수가 없다.

첫 번째 픽스는 문장 단위로 묶는 것이었다. 마침표/물음표/느낌표를 기준으로 VTT 엔트리를 그루핑해서 한 자막 블록으로 합쳤다. 타이밍은 첫 단어 시작 ~ 마지막 단어 끝으로 잡았다.

```ts
// 문장 경계 감지 후 엔트리 병합
if (/[.?!]$/.test(word.text)) {
  flushSentence(buffer);
  buffer = [];
}
```

## VTT 파싱 로직 전면 재작성

사실 파싱부터 엉망이었다. VTT 스펙에는 `NOTE`, `STYLE`, 빈 줄 구분자 등이 섞이는데, 기존 코드는 그냥 줄 단위로 읽다가 엔트리 경계에서 종종 오파싱이 났다.

파서를 처음부터 다시 썼다. 상태 머신 방식으로 `IDLE → TIMESTAMP → TEXT → FLUSH` 상태를 명시적으로 관리하니 훨씬 안정적이었다. 텍스트 정규화(연속 공백 제거, 개행 → 공백)도 이 단계에서 처리했다.

## pendingGap이 뭔데 자막을 늦췄냐

엔트리와 엔트리 사이에 갭이 있을 때(예: 00:01.200 끝, 00:01.800 시작), 이전 코드는 "다음 엔트리 시작할 때까지 기다렸다가 자막 표시"하는 pendingGap 로직이 있었다.

의도는 이해하는데, 결과는 자막 지연이었다. 갭이 0.3초만 돼도 사용자 눈엔 0.3초 늦은 자막으로 보인다.

그냥 없앴다. 갭 있으면 이전 자막 끝 타임스탬프에서 바로 끊고, 다음 자막은 다음 엔트리 시작 시각에 맞춰 등장하게.

## 마지막: 폰트 크기 85 → 76

이건 사소한데, 85가 Shorts 화면에서 너무 컸다. 긴 문장은 3줄 넘어가서 화면 절반을 잡아먹었다. 76으로 줄였더니 딱 맞다.
