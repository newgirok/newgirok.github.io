인프라를 "나중에 정리하지 뭐"라고 미루면 어떻게 되는지 우리 다 알잖아. 그래서 이번엔 처음부터 코드로 박아버렸다.

## 로컬부터 프로덕션까지 동일한 환경

제일 먼저 Docker Compose로 로컬 개발 환경을 잡았다. PostgreSQL, API(NestJS), Web(Next.js)은 당연하고, **LocalStack**을 함께 띄운 게 핵심이었다.

SQS, S3를 쓰는 파이프라인인데 개발할 때마다 실제 AWS 리소스를 쓰면 비용도 문제고 격리도 안 된다. LocalStack 하나로 SQS 큐, S3 버킷을 로컬에서 그대로 흉내낼 수 있으니까 워커 간 메시지 흐름도 로컬에서 끝까지 돌려볼 수 있다.

```yaml
# docker-compose.yml 핵심
localstack:
  image: localstack/localstack
  environment:
    - SERVICES=sqs,s3
    - AWS_DEFAULT_REGION=ap-northeast-2
```

## Terraform으로 ECS Fargate 구성

프로덕션은 ECS Fargate로 결정했다. EC2 직접 관리하기 싫고, 컨테이너 단위로 워커를 독립 배포하는 구조에 Fargate가 딱 맞았다.

Terraform 모듈을 ECS 클러스터/태스크 정의, SQS 큐, IAM 역할로 나눠서 관리한다. 나중에 워커 하나 추가할 때도 모듈 호출 한 줄이면 된다.

```hcl
module "sqs" {
  source     = "./modules/sqs"
  queue_name = "script-generation"
}

module "ecs_worker" {
  source     = "./modules/ecs"
  task_name  = "script-worker"
  queue_arns = [module.sqs.queue_arn]
}
```

IAM 역할도 최소 권한으로 Terraform이 직접 만들어준다. 콘솔에서 손으로 클릭클릭하던 시절은 이제 끝.

## GitHub Actions로 PR → 배포까지

CI/CD는 두 단계로 나눴다.

- **PR**: lint + build만. 빠르게 검증.
- **main 푸시**: ECR에 이미지 빌드 → ECS 서비스 강제 배포.

```yaml
- name: Deploy to ECS
  run: |
    aws ecs update-service \
      --cluster ${{ env.CLUSTER }} \
      --service ${{ env.SERVICE }} \
      --force-new-deployment
```

`force-new-deployment` 한 줄로 새 태스크가 뜨고 구 태스크가 내려간다. 단순하지만 지금 규모에선 충분하다.
