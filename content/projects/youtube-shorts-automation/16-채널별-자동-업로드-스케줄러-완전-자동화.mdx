## 드디어, 버튼을 누르지 않아도 된다

파이프라인을 다 만들어놓고 매번 직접 API를 날려서 잡을 생성하고 있었다. 자동화 시스템을 만들면서 정작 제일 중요한 "자동"이 빠져 있던 것. 이번에 채널별 업로드 스케줄러를 붙이면서 진짜 의미의 완전 자동화가 됐다.

## 채널마다 독립적인 cron

핵심 설계 결정은 하나였다. **채널당 독립 cron**을 쓸 것인가, 글로벌 폴링 루프 하나를 돌릴 것인가.

글로벌 루프는 단순하지만 채널별로 업로드 주기가 다를 때 유연성이 없다. A 채널은 6시간마다, B 채널은 하루에 한 번 올리고 싶은데 이걸 하나의 루프로 관리하면 결국 조건문 지옥이 된다.

`@nestjs/schedule`의 동적 cron 등록 기능을 활용했다.

```typescript
this.schedulerRegistry.addCronJob(
  `upload-channel-${channel.id}`,
  new CronJob(channel.cronExpression, () => this.triggerPipeline(channel.id)),
);
```

채널 생성/수정 시 cron을 등록하고, 비활성화 시 해제한다. DB에 `cronExpression`을 저장해두니 재시작 후에도 복원이 쉽다.

## 스케줄 실행 = 잡 생성 → 파이프라인 트리거

cron 콜백 내부는 단순하다. 잡을 DB에 생성하고 SQS에 메시지를 던지면 기존 파이프라인이 알아서 돌아간다. 스케줄러가 파이프라인 로직을 직접 알 필요가 없다는 게 포인트.

```typescript
async triggerPipeline(channelId: string) {
  const job = await this.jobService.create({ channelId });
  await this.sqsService.send('phase-0-queue', { jobId: job.id });
}
```

## 썸네일도 자동 갱신

업로드가 완료되면 YouTube API에서 썸네일 URL을 받아와 `Job.thumbnailUrl`에 저장하도록 했다. 웹 대시보드에서 업로드 결과를 확인할 때 썸네일이 바로 뜨니까 체감 품질이 확 올라간다. 작은 디테일이지만 이런 게 쌓여야 쓸 만한 도구가 된다.

## Phase 0 단독 스크립트 제거

정리하면서 `run-phase0.ts` 같은 단독 실행 스크립트들을 다 지우고 `run-pipeline.ts` 하나로 통합했다. 어차피 스케줄러가 트리거하니 단독 스크립트가 존재할 이유가 없어졌다. 코드베이스가 한결 깔끔해졌다.

---

이제 채널 설정에서 cron 표현식만 넣어주면 알아서 뉴스 가져오고, 스크립트 짜고, 영상 만들고, 업로드까지 한다. 처음 이 프로젝트를 시작할 때 그렸던 그림이 드디어 실제로 돌아가고 있다.
