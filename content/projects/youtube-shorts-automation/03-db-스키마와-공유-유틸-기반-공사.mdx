파이프라인을 만들기 전에 먼저 땅을 다져야 한다. 오늘은 여러 워커가 공통으로 쓰는 DB 스키마와 유틸 코드를 정비했다.

## Prisma 스키마 — 세 개의 모델로 시작

```prisma
model Job {
  id        String    @id @default(cuid())
  status    JobStatus @default(PENDING)
  channel   Channel   @relation(fields: [channelId], references: [id])
  channelId String
  payload   Json
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

model Channel {
  id        String             @id @default(cuid())
  ytChannelId String           @unique
  jobs      Job[]
  analytics ChannelAnalytics[]
}
```

`Job`은 파이프라인 전체의 실행 단위다. SQS 메시지를 받은 워커가 Job을 생성하고, 상태를 `PENDING → PROCESSING → DONE`으로 바꿔가며 진행 상황을 추적한다. `ChannelAnalytics`는 업로드 후 조회수/좋아요 데이터를 시계열로 쌓기 위해 따로 분리했다.

## 공유 유틸 — 워커마다 같은 코드 짜는 건 최악이니까

모노레포의 진가는 여기서 나온다. `packages/shared`에 한 번 만들어두면 NestJS API든 워커든 그냥 import해서 쓰면 된다.

핵심만 꼽자면:

**S3 래퍼** — `putObject`, `getSignedUrl` 정도만 감쌌다. 어차피 워커들이 쓰는 패턴이 거의 똑같으니까.

**환경 변수 검증** — 워커가 뜨는 순간 필수 env가 없으면 바로 죽게 만들었다. 나중에 "왜 S3 업로드가 안 돼요?"라는 삽질을 막으려고.

```ts
export function validateEnv(keys: string[]) {
  const missing = keys.filter((k) => !process.env[k]);
  if (missing.length) throw new Error(`Missing env vars: ${missing.join(', ')}`);
}
```

**pino 로거** — JSON 구조화 로그. CloudWatch에 박을 거라 사람이 읽기 편한 포맷은 포기했다.

## 솔직히

기반 공사는 재미없다. 커밋 메시지도 밋밋하고, 뭔가 만들었다는 느낌도 별로 없다. 근데 이걸 안 하면 워커 다섯 개가 각자 다른 방식으로 S3 붙고 각자 다른 방식으로 로그 남기는 지옥이 펼쳐진다. 다음 커밋부터는 진짜 파이프라인 워커를 붙일 차례다.
