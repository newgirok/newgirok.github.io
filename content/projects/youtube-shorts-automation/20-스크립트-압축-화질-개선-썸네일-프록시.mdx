오늘은 자잘해 보이지만 실제 결과물 퀄리티에 직접 영향을 주는 세 가지를 한꺼번에 손봤다.

## 스크립트 길이, 제대로 잡기

Gemini가 생성하는 스크립트가 들쑥날쑥했다. 짧으면 30초짜리 영상이 되고, 길면 90초가 넘어서 Shorts 기준(60초)을 초과해버린다. Edge TTS 기준으로 역산해보니 **210~260자가 60초에 딱 맞는 구간**이었다.

```
스크립트 길이: 210자 미만이면 재생성, 260자 초과면 트리밍
```

프롬프트에 글자 수 제한을 넣는 것만으론 부족해서, 생성 후 검증 → 범위 벗어나면 재시도하는 로직을 추가했다. Gemini가 지시를 잘 따르는 편이긴 한데, 가끔 선을 넘는다.

## FFmpeg 화질, 기본값이 너무 후졌다

초기에 렌더링 파라미터를 대충 잡아뒀더니 결과물이 뭉개졌다. CRF 값이 너무 높았고 비트레이트 제한도 없었다. Shorts는 모바일에서 소비되니까 선명도가 생명인데.

```bash
# 변경 전
-crf 28

# 변경 후
-crf 18 -b:v 4M -maxrate 6M -bufsize 8M
```

숫자 조금 바꿨을 뿐인데 체감 차이가 확실하다. 진작 할걸.

## 썸네일 CORS, 프록시로 해결

대시보드에서 YouTube 썸네일 URL을 `<img>`로 직접 참조하니까 CORS 에러가 터졌다. YouTube 이미지 서버가 cross-origin 요청을 막는 건데, 해결책은 간단하다. NestJS API 서버에 프록시 엔드포인트 하나 추가하면 끝.

```
GET /api/thumbnail?url=<encoded_youtube_thumbnail_url>
→ 서버에서 이미지 fetch 후 클라이언트로 스트리밍
```

클라이언트 입장에선 같은 오리진으로 요청하니까 CORS 걱정이 없다. 이런 거 있으면 진짜 단순하게 해결하는 게 최고다.

세 개 다 작은 변경이지만 실제 파이프라인 돌려보면 차이가 눈에 띈다. 스크립트 길이가 안정되니까 TTS → 자막 → 렌더 전 과정이 훨씬 예측 가능해졌다.
