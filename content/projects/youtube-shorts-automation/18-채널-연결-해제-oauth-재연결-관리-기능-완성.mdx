채널을 연결했으면 끊을 수도 있어야 한다. 당연한 말인데, 막상 구현하다 보면 "일단 연결만 되면 되지"하고 넘어가기 쉬운 부분이다. 이번 작업은 그 "당연한 것들"을 제대로 마무리한 주였다.

## DELETE /channels/:id — 연결 해제

API 쪽은 단순했다. `DELETE /channels/:id` 엔드포인트를 추가하고, DB에서 채널 레코드를 지우는 게 전부다. 다만 연결 해제 후 해당 채널로 예약된 업로드나 진행 중인 작업이 있으면 어떻게 할지 잠깐 고민했는데, 일단 soft delete 없이 그냥 끊는 방향으로 갔다. 복잡도 대비 실익이 없었다.

## OAuth 재연결 버그 두 개

더 재밌었던 건 재연결 관련 버그들이다.

첫 번째는 팝업 흐름 문제. OAuth 콜백 후 팝업 창을 닫아야 하는데, 콜백 URL이 `/close`를 경유하지 않고 직접 닫으려다 모바일에서 레이아웃이 깨지는 현상이 있었다. `/close` 페이지를 중간에 끼워서 `window.opener`에 메시지 보내고 닫도록 수정했다.

두 번째가 더 치명적이었다. 채널을 비활성화해뒀다가 OAuth로 재연결하면 `isActive`가 `false`인 채로 남아있는 버그. 재연결인데 비활성 상태라니... 토큰 갱신 로직에서 `isActive: true` 복원을 빠뜨린 거였다.

```ts
// OAuth 재연결 시 isActive 복원 누락 수정
await prisma.channel.update({
  where: { id: channel.id },
  data: {
    accessToken,
    refreshToken,
    isActive: true, // 이게 빠져있었다
  },
});
```

한 줄짜리 수정인데 영향은 크다. 재연결 후 업로드가 안 된다고 하면 원인 찾는 데 한참 걸렸을 거다.

## 채널 없으면 입력도 막기

웹 대시보드에서 채널이 연결되지 않은 상태로 스크립트 입력 폼이 활성화돼 있으면 사용자 경험이 이상하다. 어차피 업로드할 채널이 없으니 입력해봤자 의미가 없다. 채널 미연결 시 스크립트 입력 영역을 `disabled` 처리하고, 안내 문구를 붙였다. 작은 디테일이지만 이런 게 쌓여야 제대로 된 제품처럼 보인다.

## 채널 개설일 저장

덤으로 YouTube API의 `publishedAt` 값을 채널 개설일로 저장하도록 추가했다. 나중에 대시보드에 채널 정보 표시할 때 쓸 용도다. 지금 당장 필요한 건 아니지만, 연결 시점에 같이 받아두지 않으면 나중에 따로 API 호출해야 하니까 미리 챙겼다.
