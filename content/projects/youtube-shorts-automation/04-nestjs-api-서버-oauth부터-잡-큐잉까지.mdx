파이프라인 워커들을 하나씩 만들다 보니 어느 순간 "이걸 누가 트리거하지?"라는 질문이 생겼다. 워커들이 아무리 잘 만들어져 있어도 입구가 없으면 의미가 없다. 그래서 이번에 NestJS로 API 서버를 올렸다.

## Google OAuth부터 시작한 이유

YouTube API를 쓰려면 어차피 Google 계정 연동이 필수다. 어차피 해야 할 거, 처음부터 Passport + Google OAuth2로 인증을 잡았다. JWT로 세션 관리하는 구조라 프론트(Next.js 대시보드)에서 토큰만 들고 오면 된다.

```typescript
// google.strategy.ts
@Injectable()
export class GoogleStrategy extends PassportStrategy(Strategy, 'google') {
  constructor() {
    super({
      clientID: process.env.GOOGLE_CLIENT_ID,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET,
      callbackURL: '/auth/google/callback',
      scope: ['email', 'profile', 'https://www.googleapis.com/auth/youtube'],
    });
  }
}
```

YouTube 업로드 scope까지 한 번에 묶었다. 나중에 따로 추가하면 사용자가 재인증해야 해서 귀찮아진다.

## 채널 & 스케줄 관리

채널별로 업로드 스케줄이 다를 수 있어서 `PATCH /channels/:id/schedule`을 따로 뺐다. 채널 A는 하루 2개, 채널 B는 3일에 1개 — 이런 설정을 채널 단위로 관리한다. Prisma로 PostgreSQL에 저장하고, 스케줄러 워커가 이 설정을 읽어서 잡을 만드는 구조다.

## POST /jobs → SQS

가장 핵심은 여기다. 잡이 생성되면 바로 SQS 큐에 메시지를 던진다.

```typescript
await this.sqsService.send('script-worker-queue', {
  jobId: job.id,
  newsUrl: dto.newsUrl,
  channelId: dto.channelId,
});
```

API 서버는 메시지를 넣고 끝이다. 이후는 script-worker가 알아서 처리한다. API 서버가 워커 로직을 직접 실행하지 않으니 타임아웃 걱정이 없고, 워커가 죽어도 메시지가 큐에 남는다. SQS를 붙인 이유가 바로 이거다.

잡 상태 조회(`GET /jobs/:id`)는 Prisma로 DB를 그냥 읽는다. 워커들이 상태를 업데이트하면 API가 그걸 반환하는 구조라 실시간 폴링으로도 충분하다.
