드디어 Phase 3다. 로컬에서 `docker-compose up`으로 돌리던 걸 실제 AWS 위에 올리는 작업이다.

이 파이프라인은 서비스가 꽤 많다. API, Web, 그리고 워커만 5개 — 뉴스 수집, 스크립트 생성, TTS, 자막, 영상 렌더. 각자 역할이 다르고 스케일링 요구사항도 다르다. 이걸 한 서버에 때려박는 건 처음부터 답이 아니었고, 자연스럽게 ECS Fargate로 방향을 잡았다.

서버 관리 없이 컨테이너 단위로 독립 배포하고 스케일링할 수 있다는 게 핵심이었다. 렌더 워커는 무거우니까 태스크 크기를 키우고, 나머지는 작게 유지하는 식으로.

```hcl
module "worker_render" {
  source       = "./modules/ecs_service"
  name         = "worker-render"
  cpu          = 2048
  memory       = 4096
  image        = "${aws_ecr_repository.worker_render.repository_url}:latest"
  queue_url    = aws_sqs_queue.render.url
}
```

모듈화가 여기서 빛을 발했다. 서비스마다 `ecs_service` 모듈 하나 호출하고 cpu/memory/image만 바꾸면 된다. 반복 코드가 없으니까 실수할 여지도 줄었다.

인프라 구성 자체는 교과서적이다. ALB → Target Group → ECS Service 체인, RDS는 private subnet, SQS 큐는 워커별로 분리, S3는 중간 결과물 저장. ECR도 서비스마다 레포지토리를 따로 만들었다. 이미지 태그 관리가 훨씬 깔끔해진다.

귀찮았던 건 Security Group이다. 워커들은 외부 노출이 없는데 RDS랑 S3 접근은 허용해야 하고, ALB에서 API/Web만 트래픽 받아야 하고... 규칙 하나 빠지면 배포는 되는데 런타임에 터진다. `terraform plan` 결과 꼼꼼히 읽는 습관이 생겼다.

이제 `terraform apply` 한 방에 전체 인프라가 뜬다. 다음은 CI/CD 붙여서 푸시하면 자동 배포까지 연결하는 거다.
